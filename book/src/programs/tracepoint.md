# Tracepoints

Tracepoint programs attach to stable kernel tracepoints. Unlike kprobes,
tracepoints are part of the kernel ABI and less likely to change between
versions.

## Section Name

The section follows the format `"tracepoint/category/event"`:

```lisp
(defprog trace-fork (&key (type :tracepoint)
                          (section "tracepoint/sched/sched_process_fork"))
  0)
```

## deftracepoint

Each tracepoint has a format file under tracefs that describes its fields.
`deftracepoint` reads this format at compile time and generates typed
accessor functions automatically:

```lisp
(deftracepoint sched/sched_process_fork)
```

This generates accessors named after the tracepoint fields. For
`sched_process_fork`, these include `parent-pid`, `child-pid`, and others
matching the kernel's format file at
`/sys/kernel/tracing/events/sched/sched_process_fork/format`.

## Example: Track Process Forks

Record parent and child PIDs for every fork:

```lisp
(deftracepoint sched/sched_process_fork)

(defstruct fork-event
  (parent-pid u32)
  (child-pid u32))

(defmap events :type :ringbuf
  :max-entries (* 256 1024))

(defprog trace-fork (&key (type :tracepoint)
                          (section "tracepoint/sched/sched_process_fork"))
  (let ((e (ringbuf-reserve events (sizeof fork-event))))
    (when e
      (setf (fork-event-parent-pid e) (parent-pid ctx))
      (setf (fork-event-child-pid e) (child-pid ctx))
      (ringbuf-submit e 0)))
  0)
```

The `parent-pid` and `child-pid` accessors are generated by
`deftracepoint`. They read from the correct offsets in the tracepoint
context structure, so there is no need to manually define field offsets.
